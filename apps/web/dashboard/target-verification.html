Step 1

Choose Verification Method

Select how you want to prove control of the target. Each method generates a unique challenge token in the format sherlock_<token>.

Step 2

Verification Instructions

DNS TXT Record

Add a TXT record to the target domain:

sherlock-verification=sherlock_aBcDeFgHiJkLmNoPqRsT

Replace the token with the challenge generated by the API. DNS propagation may take minutes to hours.

Note: Real DNS checking is not implemented in this phase. Future backend will check DNS safely after SSRF protections exist.

HTML Meta Tag

Add the following tag inside <head> on the target homepage:

<meta name="sherlock-verification" content="sherlock_aBcDeFgHiJkLmNoPqRsT">

Replace the token with the challenge generated by the API.

Note: Real HTTP fetch verification is not implemented in this phase. Future backend will fetch and verify only after SSRF protections exist.

Well-Known Verification File

Create the file at:

/.well-known/sherlock-verification.txt

With content:

sherlock-verification=sherlock_aBcDeFgHiJkLmNoPqRsT

Replace the token with the challenge generated by the API.

Note: Real HTTP fetch verification is not implemented in this phase. Future backend will verify only after SSRF protections exist.

Manual Authorization Review

Provide the following information for manual review:

  • Authorization document or letter
  • Legal/scope confirmation
  • Contact person for verification
  • Allowed target(s) and scope
  • Testing window and schedule
  • Forbidden actions or exclusions

Note: File upload and admin review are not implemented in this phase. This is a placeholder for future manual authorization workflows.

Chatbot/API Challenge

Sherlock sends a safe challenge message to the target:

Return verification code exactly: sherlock_aBcDeFgHiJkLmNoPqRsT

The target API or chatbot must respond with the exact challenge string.

Note: Real API/chatbot probing is not implemented in this phase. Challenge execution must wait for network safety controls.

Security

Verification Boundaries

  • No scan without verified target

    Scanning is blocked until the target passes ownership verification.

  • No third-party target testing

    You can only verify and scan targets you own or are authorized to test.

  • No automatic crawling

    No HTTP fetch, DNS lookup, or API probing occurs until SSRF-safe controls are implemented.

  • Challenge tokens are not secrets

    Tokens prove control but do not grant access. They expire and are stored hashed if persisted.

  • Rate limiting planned

    Verification attempts will be rate-limited in future phases to prevent abuse.

  • Manual review for ambiguous targets

    Targets with unclear ownership may require manual authorization review.
