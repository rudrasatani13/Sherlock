The launch problem

AI products fail in ways normal web QA does not cover.

A chatbot can be stable and still leak system instructions. A RAG app can pass happy-path demos and still retrieve private customer data. An agent can work in staging and still misuse tools when a user injects instructions through content.

Sherlock is designed for the gap between product demo and public launch: focused testing, evidence, severity, reproduction steps, and practical fixes.

What Sherlock tests

Launch-readiness checks for modern AI surfaces.

These categories describe the planned testing surface for future audits and scanners. Phase 3 documents the methodology, but no scanner execution exists yet.

Prompt Injection

Attempts to override system intent, policy boundaries, and task constraints.

RAG Data Leakage

Checks whether retrieval can expose private, cross-tenant, or restricted data.

Agent Tool Abuse

Reviews whether tools can be invoked with untrusted or out-of-policy instructions.

Unsafe Output Handling

Looks for risky generated output, missing confirmation, and weak downstream controls.

System Prompt Leakage

Tests whether hidden instructions or security policy details are exposed.

Sensitive Data Leakage

Checks for exposure of secrets, PII, credentials, internal notes, or account context.

Indirect Injection

Tests content-borne instructions from documents, tickets, pages, and knowledge bases.

Cost Abuse

Looks for unbounded loops, excessive retries, and consumption paths that need limits.

How it works

From launch scope to retestable findings.

Sherlock's future workflow is designed to stay explicit about target ownership, tested categories, assumptions, evidence, limitations, and remediation.

Security principles

Evidence over absolute claims.

  • Sherlock should identify observed risks, not certify that an app is fully secure.
  • Reports should redact sensitive evidence and explain limitations clearly.
  • Ownership verification, SSRF protection, rate limits, and scan controls are required before public scanning.
  • API keys, logs, and customer data must be protected before any real scan execution exists.

FAQ

Practical answers before launch.

Is Sherlock a scanner or a manual audit?

PowerDetect Sherlock is positioned as an AI Launch Security Audit + Scanner. The repository currently contains a static public website, Phase 3 methodology documentation, and a Phase 4 static sample report. Future phases can add scanner logic, report systems, and manual audit workflows.

Does passing Sherlock mean my AI app is secure?

No. Passing a future scan should only mean that no blocking issues were found in the tested categories under the stated assumptions and limitations.

What types of AI products are in scope?

Chatbots, RAG systems, tool-using agents, customer-data-connected assistants, workflow copilots, and launch-critical AI features.

Will Sherlock store prompts, responses, or logs?

Real data handling is a later-phase product decision. The public security page states that scan logs may contain sensitive data and must be protected, minimized, and redacted in reports.

Is billing available?

No. The public website includes pricing and early-access positioning only. Billing, Stripe, and entitlement logic are not implemented.
